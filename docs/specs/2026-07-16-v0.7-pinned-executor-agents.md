# Design: pinned executor agents — v0.7

_Date: 2026-07-16 · Status: shipped · Author: Timo Seikel + Claude_

## 1 · Problem

Since v0.4 the workflow has had a **difficulty-graded executor**: `/implement` and `/sdd-auto` grade each
task S/M/L and dispatch S/M to sub-agents, keeping L in the main conversation (the "lead"). But the
model each tier ran on was expressed only in prose — "a subagent on a fast/small model (e.g. Haiku)",
"a general-purpose subagent on the default model". Which model actually ran was left to the lead's
per-run discretion at dispatch time. So the intended cost/capability split (cheap model for mechanical
work, mid model for standard work, the strong lead only for hard work) was a suggestion, not a
guarantee — it could silently collapse to "everything on the session model".

## 2 · Goals

1. Make the S/M tiers **deterministic**: a fixed, pinned model per tier, versioned in the plugin.
2. Keep the **lead = advisor / worker = implementer** split the plugin already embodies — the lead
   still grades, verifies, and owns all bookkeeping; workers only implement one task.
3. Don't leak the workers into non-SDD sessions as spurious auto-delegation targets.
4. Change **no contract**: no new marker phase, no new gate-allowlist entry, no new state file, no
   change to the GitHub-issue model. This is purely how the existing S/M tiers are realized.

## 3 · Decisions

- **Two plugin subagents with pinned models**, shipped in a new `agents/` directory:
  - `agents/sdd-quick.md` → `model: haiku` — the **S** (mechanical) tier.
  - `agents/sdd-standard.md` → `model: sonnet` — the **M** (standard) tier.
  - **L** stays with the lead — full context beats delegation; never delegated.
- **Model aliases, not full IDs.** `haiku` / `sonnet` resolve to the current model of each family, so
  the pin survives model refreshes without a doc edit. A personal plugin wants the family split, not a
  frozen point-in-time ID.
- **Scoped descriptions.** Each worker's `description` states it is for SDD `/implement` and `/sdd-auto`
  dispatch only, and not for general use, so the main loop doesn't auto-select them for ad-hoc work in
  unrelated repos where the plugin is installed.
- **Workers do no bookkeeping.** They carry only file/search/Bash tools (no `Task`), implement exactly
  one task, and report back. The lead verifies the result, edits the GitHub issue, moves the status
  label, and commits — identical to how the prose-dispatched sub-agents behaved before. A worker that
  finds its task is mis-graded (bigger/riskier than its tier) stops and reports for the lead to re-grade.
- **Commands dispatch by name.** `commands/implement.md` step 4 and `commands/sdd-auto.md`'s implementation
  loop now name `sdd-quick` / `sdd-standard` instead of describing a model. Same grading, same
  self-contained-prompt requirement, same lead-verifies-afterward rule.

## 4 · What did NOT change

- Contract A (the gate marker) and Contract B (the GitHub-issue model) are untouched.
- The gate-guard / session-notice hooks are untouched.
- `/implement` still does one task per gate; `/sdd-auto` still chains phases and ends at an open PR.
- The spec↔tasks consistency check, status lifecycle, and stop rules are all unchanged.

## 5 · Mirroring note

The tier→worker→model mapping now lives in four places that must stay in sync: the two worker files
under `agents/`, `commands/implement.md` step 4, `commands/sdd-auto.md`'s implementation loop, and the
executor descriptions in `CLAUDE.md` (Contract A table + auto-mode paragraph). Renaming a worker,
changing a pinned model, or adding a tier means editing all of them.
