# Design: Spec lifecycle & `/spec-cleanup` — v0.3

_Date: 2026-07-06 · Status: approved-for-planning · Author: Timo Seikel + Claude_

## 1 · Problem

Because the plugin enforces spec-driven development (one spec/plan/tasks trio per feature
under `specs/`), those artifacts pile up. Once a feature ships, its **plan** and **tasks** are
throwaway scaffolding, but nothing ever retires them — so `specs/` fills with stale plans and
task lists, and it's no longer obvious which specs describe live vs. finished work. There is
no "done" concept anywhere in v0.2.

## 2 · Goals

1. A **`/spec-cleanup`** command that archives (or, opt-in, deletes) the scaffolding of shipped
   features and flags orphaned / age-stale artifacts — conservative, transparent, human-gated,
   in the spirit of the sibling `branch-cleanup` plugin.
2. **Prevent** the pile-up at the source by giving specs an explicit **status lifecycle** that
   the existing commands maintain as work progresses, so cleanup has a reliable signal.

## 3 · Decisions (from the design interview)

- **Home:** a new command **inside the `spec-driven-development` plugin** (not a separate plugin) —
  it's tightly coupled to the `specs/` artifacts and the lifecycle edits.
- **Fate of shipped artifacts:** **archive** by default (move to `specs/archive/`), never lose
  history; `--delete` (`git rm`) is opt-in. Mirrors `branch-cleanup`'s safe-by-default stance.
- **Spec vs. scaffolding:** the **spec is a durable living doc** — it **stays in `specs/`**,
  stamped `status: done`; only **plan + tasks** move to `specs/archive/`.
- **Also in scope:** **orphans** (plan/tasks with no matching spec) and an **age-based sweep**
  (`--stale-days`, default 60) — both classified **ASK** (never auto-removed).
- **Prevention:** a **status lifecycle in spec frontmatter**, a **finalize step in `/implement`**
  (mark done + *offer* cleanup, don't auto-move), and **staleness surfaced in `/status`**.

## 4 · The status lifecycle (new convention)

Spec files gain a small YAML frontmatter block at the very top:

```yaml
---
slug: <slug>
status: draft        # draft → planned → in-progress → done
updated: 2026-07-06  # ISO date of the last status change
---
```

`status` is the one field cleanup and status rely on; `slug`/`updated` are convenience. Age is
judged primarily from git history / file mtime, with `updated` as a secondary hint.

| Value | Meaning | Set by |
|---|---|---|
| `draft` | Spec written, no plan yet | `/spec` |
| `planned` | Plan and/or tasks exist, no task done yet | `/techplan` (sets it), `/breakdown` (keeps it) |
| `in-progress` | At least one task checked, not all | `/implement` (first task checked) |
| `done` | All tasks checked / feature shipped | `/implement` (last task checked); `/reverse-spec` starts here |

- `/reverse-spec` documents already-shipped behavior → starts at **`done`** (a later `/techplan`
  for changes moves it back to `planned`).
- `/revise` that reopens a `done`/`in-progress` spec such that plan/tasks must regenerate moves the
  status back to **`planned`**.
- **Legacy specs without frontmatter** are handled by fallback signals (all-tasks-checked ⇒
  done-ish); `/spec-cleanup` offers to backfill the frontmatter.

These edits only ever touch files under `specs/`, which is on the gate-guard allowlist, so they
stay gate-safe (a `/techplan`/`/breakdown`/`/implement` run may stamp the spec's status even while
a planning gate is open).

## 5 · `/spec-cleanup` command

Args: `[--dry-run] [--delete] [--stale-days=N] [slug-filter]`.

Flow (mirrors `branch-cleanup`): gather (read-only) → classify → report → confirm → execute → summary.

**Classification:**

- **PROTECTED** (skip): the slug named by the active `.claude/sdd/phase` marker (mid-work), and any
  feature not clearly finished (status `draft`/`planned`/`in-progress` and not stale).
- **SAFE — auto-archive** (single batch confirmation): shipped features — `status: done`, or (no
  status) **all tasks checked** in `<slug>.tasks.md`. Action: move `<slug>.plan.md` +
  `<slug>.tasks.md` → `specs/archive/`, stamp the spec `status: done` (backfilling frontmatter if
  absent), and **keep the spec in `specs/`**.
- **ASK** (per-item judgment via `AskUserQuestion` for ≤4, plain list otherwise): **orphans** and
  **age-based** stale items, and abandoned in-progress features.

**Execution:** create `specs/archive/` if needed; move with `git mv` (fallback plain move) to
preserve history; `--delete` uses `git rm` instead. Never touch the active-gate slug.

**Marker:** `/spec-cleanup` **reads** `.claude/sdd/phase` (to protect the active slug) but **writes
no marker** — it is maintenance, like `/status`, though unlike `/status` it does mutate `specs/`.

## 6 · Prevention edits to existing commands

- `/spec` → write frontmatter `status: draft`.
- `/reverse-spec` → write frontmatter `status: done`.
- `/techplan` → set the spec's `status: planned`.
- `/breakdown` → ensure `status: planned`.
- `/implement` → after ticking a task: first-ever checked ⇒ `in-progress`; last one ⇒ `done` **and**
  tell the user plan+tasks are retire-able, suggesting `/spec-cleanup` (do **not** move files).
- `/revise` → move a reopened `done`/`in-progress` spec back to `planned` when downstream regenerates.
- `/status` → read `status:`, show it, and flag done/stale/orphaned features suggesting `/spec-cleanup`.
- `/next` → status-aware (recognizes `done`).

## 7 · Non-goals (YAGNI)

- No auto-move/delete from `/implement` (finalize only *offers* cleanup).
- No config file for the age threshold (a CLI flag is enough).
- No separate plugin (lives in `spec-driven-development`).
- Age-based and orphan items are never auto-removed — always ASK.
- `/spec-cleanup` is a discipline aid; like the rest of the plugin it doesn't police `Bash`.
