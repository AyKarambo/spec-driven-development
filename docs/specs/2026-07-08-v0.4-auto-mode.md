# Design: `/auto` — autonomous mode after spec approval — v0.4

_Date: 2026-07-08 · Status: shipped · Author: Timo Seikel + Claude_

## 1 · Problem

The workflow stops at a human gate after every step (`/spec` → `/techplan` → `/breakdown` →
`/implement`, one task per run). That is the right default, but once the **spec** — the artifact
that actually encodes intent — is approved, the remaining gates are often rubber stamps for
features that are well understood. There was no way to say "the spec is approved, take it from
here": every phase required a manual re-prompt, and implementation ran strictly one task per turn.

## 2 · Goals

1. A **`/auto`** command that, given an approved spec, runs the rest autonomously: technical plan,
   task breakdown, implementation, code review, and an **open PR** — no stops at intermediate gates.
2. **Enterable at any point after spec approval** — right after `/spec`, mid-planning, or
   mid-implementation. It detects existing artifacts and runs only what's missing.
3. Implementation work is **dispatched to sub-agents graded by task difficulty** (from the S/M/L
   size each task already carries).

## 3 · Decisions

- **Two human decisions are never automated.** The **spec approval** is the entry ticket:
  `specs/<slug>.spec.md` existing and non-empty *is* the approval signal, because `/spec` only
  writes the file after approval — no extra "approved" flag is needed. The **merge** is the exit:
  `/auto` always ends at an open PR, mirroring the plugin-wide "no auto-gating the merge" rule.
- **No new state.** `/auto` is a one-shot, resumable command, not a persistent mode. It introduces
  no new marker phase, no allowlist change, no config file. Resumability falls out of the
  artifacts themselves (plan/tasks files, checklist ticks, `status:` frontmatter) — the same
  signals `/status` and `/next` already read. Re-running `/auto` after an interruption continues
  from the checklist.
- **The gate-marker contract is unchanged.** Each phase run inside `/auto` follows its own
  command's marker behavior (`/techplan` writes `plan:<slug>`, `/breakdown` writes `tasks:<slug>`,
  the implementation loop deletes the marker first). The hooks need no edits; the guardrail still
  protects the planning phases even inside an autonomous run. The only difference from manual mode
  is that `/auto` doesn't *stop* at the gates.
- **Difficulty-graded executors.** S (mechanical) → sub-agent on a fast/small model; M (standard)
  → general-purpose sub-agent on the default model; L (hard/risky/cross-cutting) → the main
  conversation itself, since full context beats delegation. The orchestrator always verifies each
  result itself (tests + diff vs. the task's Check) — a sub-agent's "done" is not verification.
- **One commit per task**, so the resulting PR reads as a reviewable series.
- **Review before PR, unless already done.** The full feature diff gets a code review (native
  `/code-review` preferred, reviewer sub-agent as fallback) before `gh pr create`; skipped only
  when a review of this diff evidently already happened (review-fix commits, existing PR reviews).
- **Stop rules instead of heroics.** A task failing verification twice, a real blocker, a spec
  contradiction, or a needed scope decision stops the run with an honest report — never push
  broken or off-spec work forward. Spec **Open Questions** that materially affect the build stop
  the run at entry.
- **Branch safety.** Never runs on the default branch — it creates/switches to a feature branch
  first (it commits as it goes and ends in a PR).

## 4 · Non-goals (YAGNI)

- No persistent "auto mode enabled" flag that changes how the other commands behave — `/auto` *is*
  the mode, for exactly one run.
- No auto-merge, ever.
- No parallel task execution by default — task order encodes dependencies; parallelize only
  provably independent tasks.
- `/auto` never writes or approves the spec itself; with no approved spec it just points to `/spec`.
