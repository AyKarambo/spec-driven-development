# Design: Spec-Driven Development on GitHub Issues — v0.4

_Date: 2026-07-08 · Status: approved-for-planning · Author: Timo Seikel + Claude_

## 1 · Problem

Through v0.3 the plugin stored every spec, plan, and task list as Markdown files under
`specs/` in the consuming repo (`<slug>.spec.md`, `<slug>.plan.md`, `<slug>.tasks.md`),
with a YAML `status:` frontmatter lifecycle and a `/spec-cleanup` command whose whole job
was to keep `specs/` from filling with stale scaffolding. The artifacts are planning
by-products, not shippable code — yet they lived in the repo, cluttered diffs, and needed a
dedicated archival command to manage.

**Goal:** move specs, plans, and tasks **out of the repo and into GitHub Issues**, so the
working tree stays clean while the planning artifacts get a first-class, linkable, reviewable
home with native progress tracking.

## 2 · Decisions (from the design interview)

- **Topology:** **one GitHub issue per feature = the spec issue** (the *user story* — WHAT & WHY).
  The **technical plan** (the HOW) is an **optional `## Technical Plan` section inside the same
  spec issue body** — recommended for bigger features, skippable for small ones. **Tasks are
  native GitHub sub-issues** of the spec issue.
- **Scope:** **GitHub-issues only.** No local-file fallback. The plugin now depends on the `gh`
  CLI, an authenticated GitHub account, and a GitHub remote on the repo. Commands stop with a
  clear message when a precondition is missing.
- **Gate:** the gate-guardrail hook stays, **simplified**. There is no `specs/` allowlist entry
  anymore; while a planning gate is active, only `CLAUDE.md`, `AGENTS.md`, and `.claude/**` may be
  written. Spec/plan/task writes go to GitHub through `gh` (a `Bash` call), which the guard does
  not police — exactly the intended split.

## 3 · The GitHub model

### 3.1 Spec issue (one per feature)

- **Title:** `[SDD] <slug>: <human title>` — the `<slug>` prefix is how features are found.
- **Labels:** `sdd` (marks every SDD spec issue) **plus exactly one** status label
  (`sdd:draft` | `sdd:planned` | `sdd:in-progress` | `sdd:done`).
- **State:** open while live; **`done` ⇒ the issue is labeled `sdd:done` *and* closed.** Closing
  finished features is what keeps the issue list clean (the file-era `specs/archive/` is gone).
- **Body layout** (sections; the plan block is optional and delimited so it can be edited in place):

  ```markdown
  <!-- sdd:slug=<slug> -->
  ## User Story (Intent — WHAT & WHY)
  ## In Scope / Out of Scope
  ## Constraints
  ## Acceptance Criteria        (Given / When / Then)
  ## Resolved Questions
  ## Open Questions

  <!-- sdd:plan:start -->
  ## Technical Plan             (optional; added/filled by /techplan)
  <!-- sdd:plan:end -->
  ```

  The `<!-- sdd:slug=<slug> -->` line is a machine-readable anchor (belt-and-suspenders with the
  title). The `sdd:plan:*` comment pair lets `/techplan` and `/revise` replace just the plan.

### 3.2 Task sub-issues

- Created with `gh issue create --parent <spec#>` so GitHub tracks them as real sub-issues.
- **Title:** `[<slug>] <task title>`. **Label:** `sdd:task`.
- **Body:** `Goal:` / `Files:` / `Check:` / `Size:` (S/M/L) — the same four fields as the old
  `tasks.md` checklist item.
- **Done** = the task sub-issue is **closed**. Progress for a feature is read straight from the
  parent's `subIssuesSummary` (`{total, completed, percentCompleted}`) — no manual checkbox counting.

### 3.3 Labels (bootstrap once, idempotently)

| Label | Color | Meaning |
|---|---|---|
| `sdd` | `5319E7` | marks an SDD spec issue |
| `sdd:draft` | `BFDADC` | spec written, no plan/tasks yet |
| `sdd:planned` | `1D76DB` | plan and/or tasks exist, none done |
| `sdd:in-progress` | `FBCA04` | ≥1 task done, not all |
| `sdd:done` | `0E8A16` | all tasks done / shipped (issue also closed) |
| `sdd:task` | `C5DEF5` | a task sub-issue |

Commands ensure these exist before first use with `gh label create <name> --color <hex>
--description "…" 2>/dev/null || true` (idempotent — a pre-existing label is not an error).

## 4 · Lifecycle → labels + state

```
/spec        → create spec issue, label sdd + sdd:draft            (open)
/techplan    → fill Technical Plan section, sdd:draft → sdd:planned (open)
/breakdown   → create task sub-issues, ensure sdd:planned          (open)
/implement   → close one task; first close → sdd:in-progress;
               last close → sdd:done + close the spec issue        (closed)
/reverse-spec→ create spec issue labeled sdd:done, closed          (closed)
/revise      → reopening real work: reopen + sdd:* → sdd:planned   (open)
```

Exactly one status label is present at a time — a command adds the new one and removes the prior
one (`gh issue edit <n> --add-label … --remove-label …`).

## 5 · Slug → issue lookup

There is no repo-side mapping file (that would defeat "clean repo"). Lookup is via GitHub:

```
gh issue list --label sdd --state all --json number,title,labels,state,url,subIssuesSummary,updatedAt
```

then match the feature whose title starts with `[SDD] <slug>:` (exact slug token, to avoid
substring collisions). For a single feature, `gh issue view <n> --json body,subIssues,subIssuesSummary`
gets the full body and task list. The active gate marker still names the current `<slug>`.

## 6 · Gate marker & hooks

- **Marker unchanged in spirit:** `.claude/sdd/phase` (gitignored consuming-repo state) still holds
  one line `<phase>:<slug>` — written by planning commands, cleared by `/implement`. It is *the*
  gate signal; it does not need the issue number because lookup is by slug.
- **`gate-guard.js`** (`PreToolUse` on `Write|Edit|MultiEdit|NotebookEdit`): while the marker exists,
  deny writes outside the allowlist. **New allowlist: `CLAUDE.md`, `AGENTS.md`, `.claude/**`, and any
  path outside the repo** (`specs/**` is removed — specs are not files anymore). Spec/task writes are
  `gh` Bash calls and are intentionally not policed.
- **`session-notice.js`** (`SessionStart`): surfaces an active marker; message updated to the new
  allowlist and to say artifacts live in GitHub issues.
- **Non-negotiables preserved:** fail open on any error; zero impact when no marker; cross-platform
  Node via `${CLAUDE_PLUGIN_ROOT}`; not a security boundary (does not police `Bash`).

## 7 · Command → behavior map

| Command | GitHub behavior | Marker |
|---|---|---|
| `/constitution` | unchanged — writes `CLAUDE.md` (+ `AGENTS.md`); no issues | writes `constitution`, deletes before presenting |
| `/spec` | ensure labels; after gate, create spec issue (`sdd`+`sdd:draft`) with the user-story body | writes `spec:<slug>`, leaves it |
| `/techplan` | research; fill the `## Technical Plan` section of the spec issue; `→ sdd:planned` | writes `plan:<slug>`, leaves it |
| `/breakdown` | create one task **sub-issue** per task (`--parent`); consistency check vs acceptance criteria; ensure `sdd:planned` | writes `tasks:<slug>`, leaves it |
| `/implement` | clear marker; pick one open task sub-issue; implement+test; close it; advance spec label (`in-progress`; last → `done` + close spec) | **deletes** marker first |
| `/status` | read-only; list `sdd` issues; per-feature line from labels + `subIssuesSummary` | never touches marker |
| `/next` | same state logic; run exactly one step | delegates to the step it runs |
| `/revise` | edit the spec issue body / plan section / task sub-issues; flag downstream staleness; reopen + `→ sdd:planned` when reopening real work | writes `revise:<slug>`, leaves it |
| `/reverse-spec` | create spec issue labeled `sdd:done`, closed | writes `reverse-spec:<slug>`, leaves it |
| `/spec-cleanup` | issue hygiene: finalize finished-but-open specs (label `done` + close); flag orphan task issues & stale open specs (ASK) | reads marker to protect active slug; writes none |

## 8 · `/spec-cleanup` in the issue world

With `done` features closed automatically, GitHub already declutters the default view, so cleanup
shrinks to hygiene:

- **SAFE — finalize** (batch confirm): open spec issues whose sub-issues are **all closed**
  (`subIssuesSummary.total > 0 && completed == total`) but that aren't yet `sdd:done`/closed →
  relabel `sdd:done` + close.
- **ASK** (never automatic): **orphan** task issues (`sdd:task` with no live parent), and
  **age-stale** open specs (`updatedAt` older than `--stale-days`, default 60, not `done`).
- **Args:** `--dry-run` (report only), `--delete` (`gh issue delete` instead of close — destructive,
  per-item confirmation, opt-in), `--stale-days=N`, `slug-filter`.
- **Never** touch the active-gate slug or any non-SDD issue.

## 9 · Preconditions (every issue-touching command)

Before any `gh` work, verify and stop-with-guidance if missing:

1. `gh` on PATH and `gh auth status` succeeds.
2. The repo has a GitHub remote (`gh repo view` succeeds).

`/constitution` is exempt (writes only local rule files).

## 10 · Packaging

- Bump `plugin.json` and `marketplace.json` (plugin entry) to **`0.4.0`** — must match.
- `claude plugin validate .` must pass; `node --check` both hooks.
- README/CLAUDE.md updated: the storage story (issues, not files), the new prerequisites
  (`gh` + auth + remote), and the removal of the "fully local, no network" claim.

## 11 · Non-goals (YAGNI)

- No automatic migration of existing `specs/*.md` files into issues (a repo can adopt going
  forward; old files can be imported by hand or via `/reverse-spec`).
- No local-file fallback / dual mode.
- No GitHub Projects, milestones, or assignees automation (labels + sub-issues are enough).
- No use of the GitHub MCP server — `gh` CLI only (scriptable, no per-session OAuth, runs in `Bash`).
- Guardrail remains a discipline aid, not a security boundary (still does not police `Bash`).
