# Design: Spec-Driven Development plugin — v0.2

_Date: 2026-07-03 · Status: approved-for-planning · Author: Timo Seikel + Claude_

## 1 · Goal

Upgrade the `spec-driven-development` Claude Code plugin from v0.1 (5 German
commands) to a validated, English, installable **v0.2** that:

1. Is fully translated to English (done).
2. Polishes the 5 existing commands against the SDD principles in `research.md`.
3. Adds 4 new commands: `/status`, `/revise`, `/reverse-spec`, `/next`.
4. Adds a mechanical **gate guardrail** (hook) so planning gates can't be silently skipped.
5. Adds research subagents to `/techplan`.
6. Adds one umbrella **skill** that judges when SDD is warranted (anti over-specification).
7. Ships as a **git-repo marketplace** for install.

**Scope:** fully generic / stack-agnostic. No project-specific stack baked in — stack rules
live per-repo in `CLAUDE.md`/`AGENTS.md` (which `/constitution` generates).

## 2 · Grounding (from `research.md`)

- Full Spec Kit flow: `constitution → specify → clarify → plan → checklist → tasks → analyze → implement`.
  Gates before code: `clarify`, `checklist`, `analyze`. Lean path: `specify → plan → tasks → implement`.
- This plugin = the **lean path + constitution**, names chosen to avoid native collisions
  (`/techplan` not `/plan`=Plan Mode; `/breakdown` not `/tasks`=background jobs).
- Spec = intent + constraints + testable *Given/When/Then*, "same interpretation for every reader", living doc.
- Pitfalls: vibe-specifying, mountain of markdown, over-specifying trivial work.
- Review **informs, never auto-gates**; merge stays human.
- Some projects run Claude Code **and** Copilot → bridge is `CLAUDE.md` importing `@AGENTS.md` (§2).

## 3 · Command changes

### Existing (polish)

| Command | Maps to | Change |
|---|---|---|
| `/constitution` | `constitution` | Also emit/import **`AGENTS.md`** so Copilot + Claude share one ruleset. Sets a **transient** marker `constitution` during its turn and **clears it before presenting** (standalone setup — no downstream `/implement` to clear it). |
| `/spec` | `specify`+`clarify` | Explicit clarify pass; **record resolved questions** as a visible trail. Writes marker `spec:<slug>`. |
| `/techplan` | `plan` | Dispatch parallel **built-in `Explore` subagents** to survey the codebase before proposing the plan. Writes marker `plan:<slug>`. |
| `/breakdown` | `tasks`+`analyze` | End with a **consistency check**: every acceptance criterion maps to ≥1 task; list orphans/gaps. Writes marker `tasks:<slug>`. |
| `/implement` | `implement` | **Clears the marker first**, then implements exactly one task. Unchanged review-informs/merge-human note. |

### New

- **`/status`** — read-only. Scans `specs/` + `.claude/sdd/phase`; reports per feature
  `spec ✓ · plan ✓ · tasks ✓ · impl ▸N/M` + next step + active gate. Impl progress N/M =
  checked vs total checklist items in `specs/<feature>.tasks.md`. Writes nothing, no marker.
- **`/revise`** — update an existing spec/plan/tasks; **flag stale downstream** artifacts and
  which to regenerate. Writes only under `specs/`. Marker `revise:<slug>`.
- **`/reverse-spec`** — read existing code → produce `specs/<feature>.md` (same shape as `/spec`).
  Brownfield adoption. Never edits code. Marker `reverse-spec:<slug>`.
- **`/next`** — read state like `/status`, run the **single** next step, **stop at its gate**.
  Does not skip gates. Delegates marker handling to whichever phase it runs.

All commands keep `disable-model-invocation: true` (deliberate manual invocation).

## 4 · Gate-guardrail system

### State marker
`<repo>/.claude/sdd/phase` — a one-line file: `<phase>:<slug>` (e.g. `spec:user-login`).
- Set by planning commands as their first action; **left set** so it persists through the human gate.
- Cleared by `/implement` (and `/next`→implement) before writing code.
- Reported by `/status`; surfaced by SessionStart notice.
- Manual override: delete the file.
- Known limit: one active feature-gate at a time (documented).

### Hooks (`hooks/hooks.json`)
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit|NotebookEdit",
        "hooks": [
          { "type": "command", "command": "node \"${CLAUDE_PLUGIN_ROOT}/hooks/gate-guard.js\"" }
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          { "type": "command", "command": "node \"${CLAUDE_PLUGIN_ROOT}/hooks/session-notice.js\"" }
        ]
      }
    ]
  }
}
```

### `hooks/gate-guard.js` (Node, cross-platform)
- Reads stdin JSON: `cwd`, `tool_name`, `tool_input.file_path` (or `.notebook_path`).
- If `<cwd>/.claude/sdd/phase` **absent** → `process.exit(0)` (defer; zero impact on non-SDD work).
- If present: resolve target path; **allowlist** = `specs/**`, `CLAUDE.md`, `AGENTS.md`, `.claude/**`.
  - target in allowlist → exit 0.
  - target outside → print JSON `hookSpecificOutput.permissionDecision:"deny"` + reason
    (`🚦 <phase> gate active — approve, then /implement. Override: delete .claude/sdd/phase`), exit 0.
- Fails open on parse/IO error (never block legitimate work on a hook bug).

### `hooks/session-notice.js` (Node)
- If marker exists, emit additionalContext: `ℹ️ SDD: mid-<phase> on <slug>; artifacts gated until approved (or delete .claude/sdd/phase).` Else exit 0.

## 5 · Skill `skills/spec-driven-development/SKILL.md`
- `description` triggers when the user is about to build a substantial feature.
- Judges **whether SDD is warranted**: substantial → guide into `/constitution`?/`/spec`;
  spike/prototype → advise skipping SDD or the lean path (research pitfall #3).
- Explains the gates and lean-vs-full-vs-none. `disable-model-invocation: false`.

## 6 · Packaging & distribution

- `plugin.json`: bump to `0.2.0`, add `displayName: "Spec-Driven Development"`.
- `marketplace.json`: bump plugin entry to `0.2.0`.
- Final layout:
  ```
  spec-driven-development/
  ├── .claude-plugin/{plugin.json, marketplace.json}
  ├── commands/{constitution,spec,techplan,breakdown,implement,status,revise,reverse-spec,next}.md
  ├── skills/spec-driven-development/SKILL.md
  ├── hooks/{hooks.json, gate-guard.js, session-notice.js}
  ├── docs/specs/2026-07-03-v0.2-design.md
  └── README.md
  ```
- Validate with `/plugin validate` (or `claude plugin validate`).
- **Distribution:** host as a **git-repo marketplace**. Teammates:
  `/plugin marketplace add <repo-url>` → `/plugin install spec-driven-development@spec-driven-development-marketplace`.
- README updated: workflow diagram (9 commands + gates), guardrail explanation, when-NOT-to-use
  SDD (lean/none paths), git-repo install instructions.

## 7 · Non-goals (YAGNI)
- No auto code-review after `/implement` (rejected).
- No shipped artifact templates (rejected).
- No per-command skills (one umbrella skill only).
- No separate `/clarify` or `/analyze` commands (folded in).
- No custom research agent (built-in `Explore`).
- Guardrail is a discipline aid, not a security boundary (Bash writes not matched).
