# Design: no stray Markdown — v0.9

_Date: 2026-07-18 · Status: shipped · Author: Timo Seikel + Claude_

## 1 · Problem

Since v0.4 the spec/plan/tasks artifacts are supposed to live in **GitHub issues**, not in repo files —
that is the whole point of "the repo stays clean." But in practice the workflow could still leave
Markdown files behind, in three ways:

1. **The gate-guard had a Markdown loophole.** While a planning gate was active the guard allowed writes
   to all of `.claude/**` plus `CLAUDE.md`/`AGENTS.md`. So a spec/plan/tasks doc saved to, say,
   `.claude/specs/user-login.md` (or any Markdown that happened to match a broad allow) slipped straight
   through. And once `/implement` cleared the marker, the guard did nothing at all — nothing structural
   stopped a `spec.md` / `tasks.md` from being written.
2. **Every artifact command wrote a real `.md` temp file** (`.claude/sdd/spec-body.md`) to pipe into `gh`,
   and never deleted it. Gitignored, yes — but still an on-disk `.md` that lingered and looked like a
   stray spec.
3. **No command carried an explicit rule** that spec/plan/tasks must never be saved as files. It was
   implied by "artifacts live in issues," which the model doesn't always honor under pressure.

The user's ask: make the plugin **robust** about this — "that stuff in GitHub, no `.md` files unless it's
a general `CLAUDE.md` / constitution."

## 2 · Goals

1. A literal, easy-to-state invariant: the workflow writes **no** spec/plan/tasks Markdown. The only
   `.md` files it ever writes into a consuming repo are the project rule files — `CLAUDE.md`, `AGENTS.md`,
   and `.claude/rules/**` (the constitution).
2. Enforce it structurally where the risk is highest (the planning gates), and by explicit prompt rule
   everywhere else.
3. Preserve every hook non-negotiable: **fail open**, **zero impact when idle**, **cross-platform Node**,
   **discipline aid not a security boundary**.
4. Change **no** contract shape: no new marker phase, no new state file, no change to the GitHub-issue
   model. Only the gate-guard's rule and the body-temp-file mechanics change.

## 3 · Decisions

- **Gate-guard gains a Markdown-specific rule (marker-active only).** While the marker exists the guard
  now applies two rules:
  - **Markdown (`*.md`/`*.markdown`):** deny unless the path is `CLAUDE.md`, `AGENTS.md`, or under
    `.claude/rules/**`. This closes the loophole — a spec saved anywhere else, including under
    `.claude/`, is denied.
  - **Everything else (feature code):** unchanged — deny unless under the broad allowlist
    (`CLAUDE.md`, `AGENTS.md`, `.claude/**`), plus anything outside the repo.
- **The body temp file is no longer a `.md`.** Commands now write `.claude/sdd/issue-body.txt` (non-`.md`,
  so it passes the code rule cleanly), pipe it into `gh --body-file`, and **delete it right after**. This
  removes the last `.md` the plugin used to leave on disk and makes the invariant literal.
- **Implementation phase stays hook-free, by design.** `/implement` deletes the marker (that is still the
  moment code may be written), so during implementation the guard is idle — legitimate docs/README edits
  are never blocked. Robustness there comes from the prompt rule, not the hook. (Considered and rejected:
  keeping an `implement:<slug>` marker alive to hard-block Markdown during coding — it adds marker
  lifecycle/cleanup complexity, risks a stale marker blocking Markdown forever, and risks false positives
  on genuine documentation changes. Not worth it for the stated goal.)
- **Every artifact command carries an explicit "the issue is the only store" note.** `spec`,
  `reverse-spec`, `techplan`, `breakdown`, `revise` (planning) and `implement`, `sdd-auto` (which also
  write code) each state: spec/plan/tasks live in the issue, never in a `.md`; the only on-disk file is
  the transient non-`.md` body temp file, deleted after use.
- **The two executor agents carry the same rule.** `sdd-quick` / `sdd-standard` do the actual writing
  for S/M tasks during the implementation phase — when the marker is already deleted and the hook is idle
  by design — so the standing rule lives in their agent files too (alongside their existing "don't touch
  the issue / labels / commits" bookkeeping rule), not only in the lead-facing commands. Otherwise the
  one Write-capable surface during implementation would have neither the hook nor the prompt rule.
- **`/status` reports both gate rules.** Its gate-restriction line spells out the Markdown limit
  (`CLAUDE.md`/`AGENTS.md`/`.claude/rules/**`) separately from the broad code allowlist (`.claude/**`),
  so the readout matches what the hook actually enforces.

## 4 · What did NOT change

- Contract A's marker values and per-command marker behavior (write/leave/delete) are untouched.
- Contract B's labels, title format, body markers, and `gh` recipes are untouched (only the temp-file
  name/extension and the delete-after step changed).
- Both hooks remain fail-open, silent-when-idle, cross-platform, and non-policing of `Bash` (so `gh`
  issue writes still pass through freely).
- `/constitution` still writes `CLAUDE.md` / `AGENTS.md` / `.claude/rules/**` — those are exactly the
  sanctioned Markdown exception.

## 5 · Mirroring note

The invariant lives in several places that must stay in sync: `hooks/gate-guard.js` (the two rules),
`hooks/session-notice.js` (the notice text), `commands/status.md`'s gate-restriction line, every
artifact command's "the issue is the only store" note and its `.claude/sdd/issue-body.txt` +
delete-after wording, the two executor agents (`agents/sdd-quick.md`, `agents/sdd-standard.md`), and
`CLAUDE.md` (the top-level invariant, Contract A's gate-guard bullet, the executor-agents section, and
Contract B's "Bodies" bullet). Changing the allowed-Markdown set or the temp-file name means editing all
of them.
